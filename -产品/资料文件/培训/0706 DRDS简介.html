基本原理

DRDS 按照一定策略将数据分发到后端的每个 RDS 数据库中，这些拆分到 RDS 中的数据库被称为分库，分库中的表称为分表。

拆分键

拆分键即分库/分表字段，因此分为分库键和分表键。拆分键暂时只支持单个字段。

分库键：DRDS 根据分库键的值将数据水平拆分到后端的每一个 RDS 分库里。键值相同的数据，一定会位于同一个 RDS 数据库里。 分表键：每一张逻辑表都可以定义自己的分表键，键值相同的数据，一定会位于同一个 RDS 数据表里。

SQL路由

当用户发起执行 SQL 语句的请求时，DRDS 会理解 SQL 语句的含义，然后按照拆分键的值和执行策略将 SQL 路由到对应分区进行执行，如下图所示：

数据合并

如果一个 SQL 语句被路由到多个分表执行，DRDS 会将各个分表返回的数据按照原始 SQL 语义进行合并，并将最终结果返回给用户。

插入语句时一定要带拆分键，否则报错；

执行带有 WHERE 条件的 UPDATE、DELETE、SELECT，如带有拆分键则会分发到实际的物理库进行执行；否则会分发到所有分库上执行（即全表扫描），且执行结果会在 DRDS 中进行合并。全表扫描响应较慢，在高并发业务场景中应尽量避免使用。

原则：WHERE条件必须带分库键，尽量带分表键

表的类型

分库分表:按照策略建表，数据按照拆分键落在具体库和表（分库不分表、分表不分库、分库分表）

单库单表:普通的表，在分库0建表，数据都在分库0

广播表:在所有分库建立一样的表，数据在每个分库都由一份完全一样的（避免跨库 JOIN）

事物

DRDS：柔性事物，目前我们的DRDS版本不支持

mysql原生事物：需要确保事物内所有UPDATE、DELETE、INSERT都落在一个库，否则会报错

Sequence 介绍

DRDS 全局唯一数字序列（64 位数字，对应 MySQL 中 Signed BIGINT 类型，以下简称为 Sequence）的主要目标是为了生成全局唯一和有序递增的数字序列，常用于主键列、唯一索引列等值的生成。

目前共支持 2 种 Sequence 类型：

类型（缩写）  全局唯一    连续  单调递增    同一连接内单调递增   非单点 数据类型    可读性
Group Sequence（GROUP）   是   否   否   是   是   所有整型    好
Simple Sequence（SIMPLE） 是   是   是   是   否   所有整型    好

注：仅拆分表和广播表指定了 AUTO_INCREMENT 后，DRDS 才会创建隐式的 Sequence。且默认使用Group Sequence

重点：单库单表使用底层RDS的机制生产,是连续的;分表和广播表使用Group Sequence，是非连续的

查询原则

单表 SQL 优化有以下几个原则：

  • SQL 语句尽可能带有拆分键；（不带分库键必须评审）
  • 拆分键的条件尽可能是等值条件；
  • 如果拆分键的条件是 IN 条件，则 IN 后面的值的数目应尽可能少（需要远少于分片数，并且数目不会随业务的增长而增多）；
  • 如果 SQL 语句不带有拆分键，那么 DISTINCT、GROUP BY 和 ORDER BY 在同一个 SQL 语句中尽量只出现一种。

JOIN 优化

DRDS 的 JOIN 查询分为可下推的 JOIN 和不可下推的 JOIN（即分布式 JOIN）两类，其优化策略各不相同。 可下推的 JOIN 主要分为以下几类：

  • 单表（即非拆分表）之间的 JOIN；
  • 参与 JOIN 的表在过滤条件中均带有拆分键作为条件，并且拆分算法相同（即通过拆分算法计算的数据分布在相同分片上）；
  • 参与 JOIN 的表均按照拆分键作为 JOIN 条件，并且拆分算法相同；
  • 广播表（也称为小表广播）与拆分表之间的 JOIN。

使用 DRDS 时，应尽可能将 JOIN 查询优化成能够在分库上执行的可下

分布式 JOIN 优化

如果一个 JOIN 查询不可下推（即 JOIN 条件和过滤条件中均不带有拆分键），则需要由 DRDS 完成查询中的部分计算，即分布式 JOIN。

通常将分布式 JOIN 中的表按照数据量大小分为两类：

  • 小表：经过条件过滤后，参与 JOIN 计算的中间结果的数据量比较少（一般少于 100 条，或者相较于其它表数据更少）的表；
  • 大表：经过条件过滤后，参与 JOIN 计算的中间结果的数据量比较大（一般多于 100 条，或者相较于其它表数据更多）的表。

原则：对分布式 JOIN 的优化应将小表作为驱动表，且让驱动表带有尽可能多的过滤条件。

建表原则

  • 老表（大写）添加字段继续使用大写字段，新建表统一小写（表名和字段）

  • 数据可预见大于百万级使用分库分表，否则使用单库单表，大于5千万级的必须使用分库分表

  • 如需分库分表应与业务相关，按实际需求使用拆分键

  • 举例：

    如跟车场相关的：以车场UUID分库、时间维度分表
(如智泊云的area_parking_record 数据量大，以park_id分库、enter_time分表)

消息重发类：1消息类型分库、创建时间分表 2创建时间分库、不分表 3按时间分库、消息IDhash分表（按重发策略）

时间拆分选择 ：
            YYYYMM  按年月哈希
            YYYYWEEK    按年周哈希
            YYYYDD  按年日哈希

考虑两种场景 1、回调更新状态时，更新where条件应该是尽可能的包含拆分键
            2、扫描需要重发的数据数据时,查询where条件应该尽可能的包含拆分键

设备类：数据量小，单库单表就可以了

停车场：很多地方需要关联，应该使用广播表（很可惜、目前使用的是单库单表）
